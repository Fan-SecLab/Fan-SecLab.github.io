# 文件上传漏洞

## 一、什么是文件上传漏洞？

文件上传功能几乎是每个网站的标配——头像上传、附件提交、图片分享，处处都有上传入口。但问题在于，如果后端在处理用户上传的文件时缺乏安全意识，或者使用了有缺陷的校验逻辑，攻击者就可能通过一系列绕过手段，把恶意文件送进服务器。

一旦恶意文件被成功上传并被访问，攻击者就能拿到 Webshell，进而控制整个后台。可以说，**文件上传是获取服务器权限最快的一条路**。

## 二、测试思路

一句话：**只要看到文件上传的地方，就值得测。**

无论是头像、简历附件、图片投稿——只要允许你上传文件，就存在漏洞的可能。

### 3.2 MIME 类型校验绕过

**MIME**（Multipurpose Internet Mail Extensions）用于标识文件类型，浏览器根据它来决定如何处理文件。比如：

| 文件类型 | MIME 类型 |
|---------|-----------|
| `.jpg` / `.jpeg` | `image/jpeg` |
| `.png` | `image/png` |
| `.gif` | `image/gif` |
| `.html` | `text/html` |
| 任意二进制 | `application/octet-stream` |

每个 MIME 类型由两部分组成：大类（`image`、`audio`、`video`）加上具体种类。

---

服务端可能这样校验：

<img src="/assets/images/11/image-20260524184506620.png" alt="image-20260524184506620" style="zoom:50%;" />

抓包看看请求体：

<img src="/assets/images/11/image-20260524185025490.png" alt="image-20260524185025490" style="zoom:50%;" />

`Content-Type` 显示为 `application/octet-stream`（任意二进制），不满足图片类型的要求。**直接把 `Content-Type` 改成 `image/jpeg` 即可绕过。**

后端校验代码：

```php
if(isset($_POST['submit'])){
    $mime = array('image/jpg', 'image/jpeg', 'image/png'); // 白名单，只检查了 MIME 类型
    $save_path = 'uploads';
    $upload = upload_sick('uploadfile', $mime, $save_path);
    if($upload['return']){
        $html .= "<p class='notice'>文件上传成功</p><p class='notice'>文件保存的路径为：{$upload['new_path']}</p>";
    } else {
        $html .= "<p class=notice>{$upload['error']}</p>";
    }
}
```

**缺陷**：仅校验了 MIME 类型，对文件内容和大小没有任何限制。

> 参考：[如何使用 PHP 进行文件上传](http://www.w3school.com.cn/php/php_file_upload.asp)

### 3.3 `getimagesize()` 函数校验绕过

`getimagesize()` 是 PHP 内置函数，用于获取图像的尺寸和类型信息。成功返回数组，失败返回 `FALSE`。它会检查文件头部的标识数据，而不是只看扩展名——**这比前面两种方式靠谱得多**。

```php
array getimagesize(string $filename [, array &$imageinfo])
```

该函数支持 GIF、JPG、PNG、SWF、PSD、TIFF、BMP、IFF、JP2 等多种格式。

但它**不是把整个文件内容都检查一遍**，只检查文件头部标识格式的那几个字节。这就留下了绕过空间。

常见图片文件头标识：

```
PNG:  ‰PNG
GIF:  GIF89A
JPG:  ÿØÿà... (二进制数据)
```

#### 方法一：直接伪造头部 `GIF89A`

在 Webshell 代码的最前面加上 `GIF89A`，`getimagesize()` 就会把它当成 GIF 图片放过。

#### 🔑 关键：为什么要配合文件包含？

**但通过文件包含（File Inclusion）访问时，服务器会用 PHP 引擎解析该文件**，图片里藏着的 `<?php ... ?>` 就会被执行。

### 第四关：进阶黑名单绕过

`.phtml` 也被封了，黑名单更严。此时有三种绕过思路：

#### 法一：`.htaccess` / `.user.ini`

**`.htaccess`** 可以覆盖 Apache 的全局配置：

```apache
<FilesMatch "">
SetHandler application/x-httpd-php
</FilesMatch>
```

这样该目录下所有文件都会被当作 PHP 解析。

**前提**：
- `httpd.conf` 中 `AllowOverride ALL`（默认开启）
- 上传的文件名必须保持为 `.htaccess`（不能被改名）

效果如下：

<img src="/assets/images/11/image-20260607103805773.png" alt="image-20260607103805773" style="zoom: 33%;" />

---

**`.user.ini`** 是用户自定义的 PHP 配置文件，兼容 Apache 和 Nginx。两种用法：

在文件头自动包含：
```ini
auto_prepend_file = <filename>
```

在文件尾自动包含：
```ini
auto_append_file = <filename>
```

**前提**：
- 服务器开启 `fastcgi`（默认开启）
- 上传的文件名保持为 `.user.ini`
- 当前目录存在至少一个 `.php` 文件
- PHP 版本 ≥ 5.3

**实战**：在 `.user.ini` 中写入 `auto_prepend_file = 111.jpg`，然后在 `111.jpg` 中写入 `<?php phpinfo(); ?>`：

### 第八关：NTFS 数据流（`::$DATA`）

Windows NTFS 文件系统支持备用数据流（Alternate Data Stream）：

```
1.txt:stream:$DATA
```

- `1.txt`：宿主文件
- `stream`：隐藏的附加数据流
- `$DATA`：NTFS 固定流类型

这个附加流的内容在资源管理器中完全不可见，属于 Windows 特有的隐藏机制。可以结合文件上传场景绕过黑名单。

---

### 第九关：末尾点号 + 空格组合

既过滤了末尾点号，也过滤了空格，但可以组合：`webshell.php. .`

### 第十一关：`%00` 截断（GET）

上传 `.php` 并通过 MIME 绕过成功：

![image-20260708180408518](/assets/images/11/image-20260708180408518.png)

<img src="/assets/images/11/image-20260708180445987.png" alt="image-20260708180445987" style="zoom:33%;" />

但访问图片路径时发现**文件名被后端改掉了**：

<img src="/assets/images/11/image-20260708181157338.png" alt="image-20260708181157338" style="zoom: 50%;" />

观察请求：`save_path=../upload/` 决定了保存路径，后端再拼接随机数 + 日期作为新文件名。

**绕过思路**：在 `save_path` 中插入 `%00`：

```
save_path=../upload/webshell.php%00
```

`%00` 是 URL 编码的空字符（NULL），在截断有效的环境下，`%00` 后面的所有内容会被忽略，从而保留 `webshell.php` 这个文件名。

**前提**：PHP < 5.3.4，且 `magic_quotes_gpc = OFF`。

上传成功后访问：

![image-20260708182522940](/assets/images/11/image-20260708182522940.png)

如果访问完整路径出现 `%EF%BF%BD`（URL 解析后的替换字符），直接去掉后面部分，访问 `http://xxx/upload/upload/webshell.php` 即可。

后端核心代码分析：

```php
$file_ext = substr($_FILES['upload_file']['name'], strrpos($_FILES['upload_file']['name'], ".") + 1);
// 取最后一个 . 之后的内容作为后缀

if(in_array($file_ext, $ext_arr)){
    $temp_file = $_FILES['upload_file']['tmp_name'];
    $img_path = $_GET['save_path'] . "/" . rand(10, 99) . date("YmdHis") . "." . $file_ext;
    // 拼接路径 = GET 参数的 save_path + 随机数 + 时间戳 + 后缀

    if(move_uploaded_file($temp_file, $img_path)){
        $is_upload = true;
    }
}
```

### 第十七关：条件竞争（Race Condition）

核心逻辑：

```php
if(move_uploaded_file($temp_file, $upload_file)){
    if(in_array($file_ext, $ext_arr)){
        $img_path = UPLOAD_PATH . '/' . rand(10, 99) . date("YmdHis") . "." . $file_ext;
        rename($upload_file, $img_path);
        $is_upload = true;
    } else {
        $msg = "只允许上传.jpg|.png|.gif类型文件！";
        unlink($upload_file); // 删除文件
    }
}
```

文件先被上传到服务器，然后检查后缀——不合格就删除。但在**上传完成和删除之间有一个微小的时间窗口**。

**攻击思路**：在这个时间窗口内访问 Webshell，让它生成一个新的 Shell 文件。

准备 `haha.php`：

```php

$myfile = fopen("jaden.php", "w");
$txt = "<?php phpinfo();?>";
fwrite($myfile, $txt);
fclose($myfile);
?>
```

### 第二十关：数组绕过

代码逻辑分析：

```php
$file = empty($_POST['save_name']) ? $_FILES['upload_file']['name'] : $_POST['save_name'];
if (!is_array($file)) {
    $file = explode('.', strtolower($file));
    // 按 . 拆分，全部转小写
}

$ext = end($file);          // 取最后一个元素作为后缀（白名单校验点）
$file_name = reset($file) . '.' . $file[count($file) - 1];
// 拼接：第一个元素 + . + 最后一个元素
```

问题在于：**可以通过 POST 传入数组，使 `explode()` 跳过，直接使用数组结构。**

上传时抓包，构造多个 `save_name[]` 参数：

```
save_name[0]=1.php
save_name[2]=
save_name[3]=
save_name[4]=jpg
```

![image-20260709233127340](/assets/images/11/image-20260709233127340.png)

- `end($file)` → `jpg`，通过白名单检查 ✅
- `reset($file)` → `1.php`
- `$file[count($file) - 1]` → `save_name[3]` 为空，拼接结果变成 `1.php.`

末尾的点号不影响文件执行，绕过成功！

## 五、总结

文件上传漏洞的攻防是一个持续的博弈。回顾整个靶场，防御手段从易到难：

| 关卡 | 防御手段 | 绕过方式 |
|------|---------|---------|

### 防护建议

作为开发者或安全运维，要做好文件上传防护，记住以下几点：

1. **不要信任任何用户输入**——前端校验只是 UX，不能作为安全手段。
2. **后端使用白名单**，白名单比黑名单更可靠。
3. **校验文件内容而非仅校验扩展名**——结合 MIME 类型、文件头、文件大小等多维度判断。
4. **重命名上传文件**——避免保留用户原始文件名。
5. **限制上传目录的执行权限**——即使文件上传成功，也不让服务器解析执行。
6. **使用独立的文件存储服务**（如 OSS），与 Web 服务隔离。
7. **及时更新 PHP 版本和中间件**，修复已知的解析漏洞。

文件上传漏洞是 Web 安全领域的"入门课"，但也是"必修课"。希望这篇笔记能帮你建立起系统的攻防思维。
