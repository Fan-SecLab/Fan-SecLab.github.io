## 一、XSS 简介

**XSS** 全称 Cross Site Scripting（跨站脚本攻击），为了避免与 CSS 层叠样式表名称冲突，故缩写为 XSS。

XSS 是指攻击者在网页中嵌入客户端脚本（通常是 JavaScript 编写的恶意代码），当用户浏览网页时，脚本在用户的浏览器上执行，从而达到攻击目的。

> XSS 属于**客户端攻击**，受害者最终是用户。网站管理员也属于用户之一，这意味着 XSS 可以进行"类服务端"攻击——因为管理员权限远大于普通用户，攻击者通常以管理员身份作为跳板实施进一步攻击。

XSS 攻击的最终目的：在网页中嵌入客户端恶意脚本代码，最常用的攻击语言是 **JavaScript**。

## 二、XSS 出现的原因

程序对**输入和输出的控制不够严格**，导致精心构造的脚本输入后，在输出到前端时被浏览器当作有效代码解析执行，从而产生危害。

简单来说：**用户输入了不该输入的东西，程序没拦住，浏览器当真执行了。**

---

## 三、XSS 的危害

| 危害类型 | 说明 |
| :--- | :--- |
| **劫持用户 Cookie** | 窃取会话凭证，冒充用户身份 |
| **框架钓鱼** | 利用 JS 操作 DOM，生成虚假页面欺骗用户输入敏感信息 |
| **键盘记录** | 监听用户按键，记录密码等输入内容 |
| **网页篡改** | 修改页面内容、插入恶意链接 |
| **传播蠕虫** | 通过 XSS 自动向其他用户发送恶意请求 |

**攻击流程：**

### 4.3 DOM 型 XSS

#### DOM 简介

浏览器将网页（HTML）整理成一棵**节点树**，让 JS 代码能够读取和修改页面内容、结构和样式。

**DOM 树结构：**

![DOM树](/assets/images/08/image-14.png)

DOM 全称 Document Object Model（文档对象模型），是与系统平台和编程语言无关的接口，程序和脚本可以通过它动态地访问和修改文档内容、结构和样式。

> **关键点：** DOM 型 XSS 的产生原因是**前端 JS 代码的 DOM 操作**，不与后端发生交互。

**可能触发 DOM 型 XSS 的 JS 操作：**

| 属性/方法 | 作用说明 |
| :--- | :--- |
| `document.referrer` | 获取进入当前页面的来源网址，常被用来窃取来路信息 |
| `window.name` | 浏览器窗口全局名称，页面跳转后值保留，可传递恶意代码 |
| `location` | 操作当前页面 URL、链接参数，DOM XSS 最主要的参数来源 |
| `innerHTML` | 将内容解析成 HTML 标签渲染，不过滤直接写入会触发 DOM XSS |
| `document.write` | 向页面直接输出内容，可写入 HTML 和 JS，存在 XSS 风险 |

#### 示例一：`innerHTML` 注入

**Payload：**

```html
' onclick="alert(1111)"
```

```html
' onclick="alert('xss')">
```

```html
'><img src="#" onmouseover="alert('xss')">
```

**效果：**

![示例一效果](/assets/images/08/image-17.png)
![示例一弹窗](/assets/images/08/image-18.png)

**原理分析：**

提交正常数据时，访问的是一个相对路径：

```html
<a href='jaden'>what do you see?</a>
```

浏览器把 `jaden` 解析为相对路径，点击 `what do you see?` 会尝试跳转。

**网站源代码：**

HTML 结构：

```html
<input id="text" name="text" type="text" value>
<input id="button" type="button" value="click me!" onclick="domxss()">
<div id="dom"></div>
<a href='jaden'>what do you see?</a>
```

- `<input id="text">` — 接收用户输入
- `<input id="button">` — 点击执行 `domxss()` 函数
- `<div id="dom">` — 展示结果

JS 核心漏洞函数：

```javascript
function domxss() {
  var str = document.getElementById("text").value;
  document.getElementById("dom").innerHTML = "<a href='" + str + "'>what do you see?</a>";
}
```

**代码逐行解析：**

| 代码 | 作用 |
| :--- | :--- |
| `document.getElementById("text")` | 根据 id 找到输入框 |
| `.value` | 获取输入框中用户输入的内容 |
| `var str = document.getElementById("text").value` | 将用户输入保存到变量 `str` |
| `document.getElementById("dom")` | 根据 id 找到 div 容器 |
| `.innerHTML` | 往标签里写入内容，会解析成 HTML 标签执行 |
| `"<a href='" + str + "'>what do you see?</a>"` | 拼接字符串，用用户输入作为链接地址 |

> **漏洞本质：** `innerHTML` 会将拼接内容当作 HTML 解析执行。输入 `jaden` 正常显示链接，但输入恶意 JS 代码则会被执行。

**正常输入流程：**

1. 获取输入框内容：`str = "jaden"`
2. 字符串拼接：`"<a href='jaden'>what do you see?</a>"`
3. 渲染显示：页面出现文字链接 **what do you see?**

**总结：** 把输入框的内容变成一个链接地址显示在页面上，`innerHTML` 负责解析执行。

## 五、XSS 可能出现的位置

通过源码判断是否存在漏洞：

### 1. HTML 页面直接输出

含有 `<p>...</p>` 等标签，将用户输入直接呈现在页面上：

![直接输出](/assets/images/08/image-37.png)

### 2. 标签属性输出

```html
<img src="image.png" alt="{{Untrusted Data}}">
<input type="text" value="{{Untrusted Data}}">
<body onload="{{Untrusted Data}}">
```

`{{Untrusted Data}}` 是占位符，表示用户输入会被直接拼进属性值中。

### 3. URL 相关属性

```html
<iframe src="{{Untrusted Data}}">
<a href="{{Untrusted Data}}">Link</a>
<META http-equiv="refresh" content="5,URL={{Untrusted Data}}">
```

**常见 Payload：**

```html
" onload="alert(1)
```

### 手工检测思路

1. 在目标站点上找到**输入点**（查询接口、留言板等）
2. 输入一组 **"特殊字符 + 唯一标识字符"**，提交后查看源码是否做了处理
3. 结合唯一字符前后的语法或函数，确认是否可以构造闭合条件执行 JS

> **核心原则：** 考虑哪里有输入，输入的数据在什么地方输出。只要有用户输入输出的地方，都可能存在 XSS 漏洞。

| 片段 | 作用 |
| :--- | :--- |

| `document.cookie` | 获取当前用户的 Cookie |
| `+` | 拼接 Cookie 值作为 URL 参数 |

整个 payload 的含义：**将 Cookie 作为 URL 参数发送到攻击者的服务器。**

**环境搭建：**

- 漏洞服务器：修改 `config.inc.php` 配置后初始化 Pikachu 靶场

![环境搭建](/assets/images/08/image-39.png)
![后端代码](/assets/images/08/image-40.png)

**攻击过程：** 找到存在存储型 XSS 的留言板 → 写入 payload → 用户访问时 Cookie 被发送到恶意服务器 → 用户自动跳转回正常页面。

**rk.js 核心函数：**

| 函数 | 作用 |
| :--- | :--- |
| `createAjax()` | 自动适配浏览器，悄悄向恶意服务器发送数据（无需刷新页面） |
| `show()` | 接收后端返回的数据，动态渲染到页面元素上 |
| **`onkeypress()`** | **重点** — 按键按下时触发的函数 |

**按键捕获原理：**

```javascript
var realkey = String.fromCharCode(event.keyCode);
```

- `event.keyCode` — 将按键转换为 ASCII 码
- `String.fromCharCode(event.keyCode)` — 将 ASCII 码转换回对应字符
- `xl += realkey` — 拼接到记录变量中

> 浏览器只认识 ASCII 编码，不认识字母，所以需要 `String.fromCharCode()` 转换。

**工作流程：**

```
按键触发 onkeypress()
  → 调用 createAjax() 偷偷发送数据
  → 后端返回确认
  → createAjax 传给 show()
  → show 将内容渲染到页面
```

## 十二、编码绕过

### 编码基础

**浏览器解析顺序：** URL 解析器 → HTML 解析器 → CSS 解析器 → JS 解析器

**URL 解码的特殊性：** URL 解码在后台检测之前自动执行，即后台收到 URL 后会先解码，再执行过滤函数。GET 方式提交时如果数据中包含 HTML 实体编码（含有 `&`、`#` 等字符），需要对这些特殊字符进行 URL 编码以保证正常解析。POST 方式则不需要。

**HTML 实体编码规则：**

1. 只能改属性值/文本内容，不能碰标签名、属性名、语法符号
2. 不能破坏 DOM 树结构
3. 只在 HTML 页面、标签属性里生效，进入 JS 环境后失效

**JS 编码规则：**

1. 只能改普通字符，不能碰括号和引号
2. 只在 JS 解析环境内才解码
3. JS 环境中不解析 HTML 实体
4. **特殊情况：** 在 `javascript:` 伪协议中，既是 HTML 属性值（能被 HTML 解析器解实体编码），又是 JS 代码（能被 JS 解析器解码），**双重解析，两种编码都能用**

**数据提交流程：**

```
输入 payload → 提交到后台 → 后台过滤处理 → 返回给浏览器 → 浏览器解析 HTML
```

**核心思路：** 后台可能对关键字进行过滤（先过滤再返回），我们可以尝试将关键字编码后提交，浏览器解码后还原为正常代码。关键是判断**编码在输出时是否会被正常识别和翻译**。

**在线编码工具：**

- [ASCII 编码工具](http://tools.gree020.cn/ascii/)
- [Unicode 编码工具](https://tool.chinaz.com/tools/unicode.aspx)
- [编码解码工具](https://www.toolhelper.cn/EncodeDecode/UnicodeChinese)

## 十五、XSS 之 HttpOnly 绕过

### HttpOnly 是什么

`HttpOnly` 是包含在 HTTP 响应头 `Set-Cookie` 中的一个附加标志。设置后，客户端脚本（JavaScript）将**无法读取**该 Cookie。

## 十六、XSS 常规防范

**总体思路：对输入进行过滤，对输出进行编码。**

| 措施 | 说明 |

## 十八、同源策略与跨域

### 什么是跨域

```
http:// www. jaden.cn :80 /news/index.php
协议   子域名 主域名   端口   资源地址
```

当**协议、主机（主域名 + 子域名）、端口**中任意一个不相同时，称为**不同域**。不同域之间请求数据的操作叫做**跨域操作**。

```
http://www.jaden.cn:80   ← 同一域
http://www.jaden.cn:81   ← 端口不同，不同域
https://www.jaden.cn:80  ← 协议不同，不同域
http://www.jadens.cn:80  ← 域名不同，不同域
```

### 同源策略

为了安全，所有浏览器都约定了**同源策略**：不同域之间不能使用 JS 进行操作。比如 `x.com` 域名下的 JS 不能操作 `y.com` 域名下的对象。

> **为什么要有同源策略？** 假设没有同源限制，恶意网页可以通过 JS 嵌入银行登录页面，在用户登录时获取用户名和密码。

以下标签的跨域加载**不受**同源策略限制（但资源类型有限制）：

```html
<script src="...">   <!-- 加载 JS 执行 -->
<img src="...">      <!-- 加载图片 -->
<link href="...">    <!-- 加载 CSS -->
<iframe src="...">   <!-- 加载任意资源 -->
<a href="...">       <!-- 超链接 -->
```

### JSONP 跨域流程总结

1. **声明回调函数** — 等待后端调用，接收数据
2. **动态创建 `<script>` 标签** — 利用 `<script>` 的 `src` 天生允许跨域的特性发起请求
3. **URL 携带回调函数名** — 如 `?cb=fn`，告知服务端以什么函数名返回数据
4. **服务端拼接并返回 JS 代码** — 如 `fn('username=xxx&password=yyy')`
5. **浏览器自动执行返回的 JS 代码** — 回调函数被调用，数据到手
